Three approaches to master pump zone + time-block scheduling

My recommendation is A — here's why, and what the alternatives cost you.

A

Device-level pump zone (recommended)

One setting on the device: "Zone 1 is the pump." Every program automatically wraps its queue steps with a pump-on before the first zone and a pump-off after the last. Back-to-back programs skip the off/on between them — pump runs continuously through the whole irrigation window.

  • DB change: add pump_zone_num to existing farm_devices table
  • UI change: one selector in device settings (Admin Console)
  • Scheduler change: run-schedules Edge Function injects pump steps automatically
  • Multiple programs share the same pump zone — consistent, simple
B

Per-program pump zone

Each program (zone group) independently declares which zone is its pump. Avocados might use Zone 1, Citrus might use Zone 1 too — or a different relay for a different block.

  • DB change: add pump_zone_num to zone_groups
  • UI change: pump selector in each program's settings
  • Scheduler: each program builds its own pump steps independently
  • ⚠ Back-to-back pump coordination harder — two programs must detect overlap
C

Pump as an explicit zone step

No special "pump" concept in the DB. The user manually adds Zone 1 as the first and last step in each program, with duration spanning the whole block. The app treats it like any other zone.

  • No DB change needed
  • Full flexibility — pump step sits wherever the user places it
  • ⚠ User must remember to add it to every program
  • ⚠ No automatic back-to-back coordination — pump cycles off then on between programs

What Approach A builds — concretely

DB changes (small)

  • farm_devices.pump_zone_num — nullable int
  • No changes to zone_groups or program_queue

App changes

  • Admin Console: pump zone selector per device
  • Program builder: per-zone duration inputs (replaces global duration)
  • run-schedules: inject pump on/off steps in queue
  • Calendar: show pump zone as continuous bar across blocks
